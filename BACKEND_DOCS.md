# Backend Documentation (All 3 Services)

This document describes the backend contract for all three services in this repository:

- upload-service (HTTP API that accepts repo URL and queues build jobs)
- deploment-service (background worker that builds and publishes artifacts)
- request-handler-service (HTTP service that serves built files by subdomain)

## 1) High-level Flow

1. Client calls upload-service POST /deploy with a Git repository URL.
2. upload-service clones repo, uploads source files to S3 under output/{id}/..., pushes id to Redis list build-queue, and marks Redis status[id]=uploaded.
3. deploment-service pops id from Redis build-queue, downloads source from S3, runs npm install and npm run build, uploads dist files to S3 under dist/{id}/..., and marks Redis status[id]=deployed.
4. request-handler-service reads subdomain from host, maps it to id, and serves files from S3 dist/{id}/... .

## 2) Shared Infrastructure Contracts

### Redis keys and values

- List key: build-queue
  - Producer: upload-service uses LPUSH build-queue {id}
  - Consumer: deploment-service uses BRPOP build-queue 0
- Hash key: status
  - upload-service sets status[{id}] = uploaded
  - deploment-service sets status[{id}] = deployed

### S3 key patterns

- Uploaded source: output/{id}/{relativePath}
- Built artifacts: dist/{id}/{relativePathInsideDist}

### ID format

- Generated by upload-service generateId()
- 5 characters
- lowercase a-z and digits 0-9

## 3) upload-service

- Folder: upload-service
- Default port: 3000
- Base URL (local): http://localhost:3000

### Required environment variables

- AWS_ACCESS_KEY_ID
- AWS_SECRET_ACCESS_KEY
- S3_BUCKET_NAME
- S3_BUCKET_REGION

### Route: POST /deploy

Creates a deployment job from a Git repository URL.

#### Request

- Method: POST
- Path: /deploy
- Headers: Content-Type: application/json
- Body:

```json
{
  "repoUrl": "https://github.com/<owner>/<repo>"
}
```

#### Success response

- Status code: 200
- Body:

```json
{
  "id": "abc12",
  "message": "Deployment successful"
}
```

#### Error responses

- Status code: 500
- Body:

```json
{
  "message": "Internal server error"
}
```

#### Known error triggers

- Invalid or missing repoUrl (clone failure)
- Git clone/network/auth failures
- Missing AWS env vars (throws inside uploadFile)
- Redis command failures
- Any uncaught runtime error in handler

#### Notes about current behavior

- There is no explicit request validation for repoUrl before clone.
- uploadFile logs S3 upload errors internally; it does not rethrow from inside its try/catch.
- Because of that, partial upload failures can still result in HTTP 200 and message Deployment successful.

### Route: GET /status/:id

Returns deployment status from Redis hash status.

#### Request

- Method: GET
- Path: /status/:id
- Path params:
  - id: deployment id returned by POST /deploy

#### Resolver flow

1. Read id from req.params.id.
2. Fetch Redis hash value hGet(status, id).
3. Return JSON with id and status.

#### Success response

- Status code: 200
- Body example when known:

```json
{
  "id": "abc12",
  "status": "uploaded"
}
```

- Body example after build:

```json
{
  "id": "abc12",
  "status": "deployed"
}
```

- Body example when id not found:

```json
{
  "id": "abc12",
  "status": null
}
```

#### Error responses

- No explicit error handling in route.
- If Redis call throws, Express default error handling applies (typically 500).

## 4) deploment-service (build worker)

- Folder: deploment-service
- Type: background worker (no HTTP routes)
- Default port: none

### Required environment variables

- AWS_ACCESS_KEY_ID
- AWS_SECRET_ACCESS_KEY
- S3_BUCKET_NAME
- S3_BUCKET_REGION

### Route surface

- HTTP routes: none
- External contract: Redis queue consumer + S3 build pipeline

### Worker resolver flow

1. Connect Redis subscriber and publisher clients.
2. Loop forever.
3. BRPOP from build-queue (blocking).
4. For popped id:
   - Download source folder from S3 prefix output/{id}
   - Build project in local output/{id} using shell command:
     - npm install
     - npm run build
   - Copy local output/{id}/dist recursively to S3 prefix dist/{id}/
   - Set Redis hash status[id] = deployed

### Worker output/state transitions

- Input status (from upload-service): uploaded
- Output status (from worker): deployed

### Error behavior

- Missing AWS env vars in download/upload helper throws Error("Missing AWS environment variables").
- There is no top-level try/catch around the processing loop.
- A thrown error can stop the worker process.
- Build command exit code is not checked for failure; buildProject resolves on process close regardless of success/failure.
- If dist folder is missing or unreadable, recursive file discovery returns empty list and worker can still mark status as deployed.

## 5) request-handler-service

- Folder: request-handler-service
- Default port: 3001
- Base URL (local): http://localhost:3001

### Required environment variables

- AWS_ACCESS_KEY_ID
- AWS_SECRET_ACCESS_KEY
- S3_BUCKET_NAME
- S3_BUCKET_REGION

### Route: catch-all (all paths)

This service uses one catch-all Express middleware route.

#### Request resolution rules

1. Read host from req.hostname.
2. Resolve deployment id as first segment of hostname:
   - id = req.hostname.split(".")[0]
3. Read request path req.path.
4. Build S3 key:
   - If path is /, use index.html
   - Else remove leading / and convert forward slash to backslash
   - Final key: dist/{id}/{resolvedPath}

#### Why slash conversion exists

The implementation converts / to \\ to match Windows-style separators in uploaded S3 keys for nested files.

#### Success response

- Status code: 200
- Body: streamed file content from S3
- Content-Type is inferred from requested path extension:
  - .html -> text/html
  - .css -> text/css
  - .js -> application/javascript
  - .svg -> image/svg+xml
  - .png -> image/png
  - .jpg/.jpeg -> image/jpeg
  - .ico -> image/x-icon
  - fallback -> application/octet-stream

### SPA fallback resolver

If requested key is missing with S3 NoSuchKey:

1. Try to fetch dist/{id}/index.html
2. If found, return 200 text/html with fallback file
3. If fallback also missing/fails, return 404 Not found

### Error responses

- Status code: 404
  - Body: Not found
  - Trigger: requested key missing and fallback index missing

- Status code: 500
  - Body: Internal server error
  - Trigger: non-NoSuchKey S3 failure or runtime error in fetch branch

## 6) Status Code Matrix

- upload-service
  - POST /deploy: 200, 500
  - GET /status/:id: 200 (and implicit 500 on unhandled Redis failure)
- request-handler-service
  - catch-all file fetch: 200, 404, 500
- deploment-service
  - no HTTP status codes (background worker)

## 7) Error Catalog (Consolidated)

- Missing AWS env vars
  - upload-service: becomes 500 in /deploy catch path
  - deploment-service: can throw and stop worker loop
  - request-handler-service: logs at startup; later S3 usage may fail and return 500

- Redis unavailable
  - upload-service: connect errors logged at startup; command-time failures can cause 500 or unhandled route errors
  - deploment-service: connect failure prevents job processing

- Git clone/build failures
  - upload-service clone failure: 500 from /deploy
  - deploment-service build failure: currently may still continue and set deployed due to missing exit-code validation

- S3 upload/download failures
  - upload-service file upload failures can be logged but not propagated per-file
  - deploment-service upload/download issues may throw or log depending on helper path

## 8) Practical API Usage Examples

### Trigger deployment

```bash
curl -X POST http://localhost:3000/deploy \
	-H "Content-Type: application/json" \
	-d '{"repoUrl":"https://github.com/<owner>/<repo>.git"}'
```

### Poll deployment status

```bash
curl http://localhost:3000/status/abc12
```

### Request deployed asset (host-based id resolution)

Example host where subdomain is deployment id abc12:

```bash
curl -H "Host: abc12.localhost" http://localhost:3001/
curl -H "Host: abc12.localhost" http://localhost:3001/assets/main.js
```
